LAB-001: Securing IoT Device Communication with TLS/SSL

[ SOURCE ] CND coursework — Bevywise MQTT Broker + IoT Simulator (Windows VMs)

[ THIS LAB ] Same lesson, ported to Docker on Ubuntu — no Windows VMs required
$ lab read LAB-001 --section=overview
[ overview ]
MQTT is the default messaging protocol for IoT — lightweight, publish/subscribe, built for low-bandwidth machine-to-machine traffic. It is also, by default, completely unauthenticated and unencrypted. Anyone who can see the wire can read every topic and every payload. This lab demonstrates that exposure directly, then closes it with TLS.
The original CND coursework version of this lab runs on two Windows VMs (a broker server and an admin/client machine) using the commercial Bevywise CrystalMQ broker and IoT Simulator, with Wireshark doing the packet capture. None of that tooling is Linux-native, so this version substitutes the open-source equivalent at every step and gets the identical result — arguably a stronger one, since the TLS handshake here negotiated TLS 1.3, one version ahead of the original lab's TLS 1.2.

All three run as separate Docker containers on a user-defined bridge network (iot-lab-net), mirroring the two-VM separation in the original lab. A fourth container, sharing the broker's network namespace, does the packet capture — this is the trick that avoids needing sudo on the host: Docker itself already has the required capabilities, and the capture container inherits them.
$ lab read LAB-001 --section=phase1
[ phase 1 — unencrypted, port 1883 ]
Broker config for this phase is deliberately minimal — the default, insecure state most IoT devices ship in:
listener 1883 allow_anonymous true
Bring up the broker and a subscriber standing in for the IoT device:
# the broker docker run -d --name mqtt-broker --network iot-lab-net \ -v "$(pwd)/mosquitto/mosquitto-notls.conf:/mosquitto/config/mosquitto.conf" \ eclipse-mosquitto # the "device" — subscribes to the same topic the original lab used docker run -d --name iot-device --network iot-lab-net eclipse-mosquitto \ mosquitto_sub -h mqtt-broker -p 1883 -t High_Tempe -v
Start the capture inside the broker's own network namespace, then send the command from a third container standing in for the web server:
# capture — no sudo needed docker run -d --rm --name pkt-capture \ --network container:mqtt-broker \ --cap-add=NET_ADMIN --cap-add=NET_RAW \ -v /tmp/capture:/capture \ nicolaka/netshoot \ tcpdump -i any -A -s0 -w /capture/phase1-notls.pcap 'tcp port 1883' # the "server" sends the alert docker run --rm --network iot-lab-net eclipse-mosquitto \ mosquitto_pub -h mqtt-broker -p 1883 -t High_Tempe -m "Alert for High Temperature"
evidence — raw capture, grep for the payload
$ tcpdump -r phase1-notls.pcap -A | grep "High_Tempe\|Alert for" High_TempeAlert for High Temperature
Both the topic name and the message body are sitting in the capture as plain readable text. Anyone positioned on this network segment — a compromised switch, a rogue AP, a malicious neighbor on the same subnet — reads this exactly as easily as the device did.
$ lab read LAB-001 --section=phase2
[ phase 2 — TLS enabled, port 8883 ]
Generate a CA and a server certificate. The server cert's CN has to match the hostname clients actually connect to — mqtt-broker, the Docker service name, not a real DNS name, since this never leaves the bridge network:
openssl genrsa -out ca.key 2048 openssl req -x509 -new -nodes -key ca.key -sha256 -days 365 \ -out ca.crt -subj "/C=US/ST=Lab/L=Lab/O=CND-IoT-Lab/CN=iot-lab-ca" openssl genrsa -out server.key 2048 openssl req -new -key server.key -out server.csr \ -subj "/C=US/ST=Lab/L=Lab/O=CND-IoT-Lab/CN=mqtt-broker" openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial \ -out server.crt -days 365 -sha256
Broker config now requires TLS on 8883 instead of plain 1883:
listener 8883 allow_anonymous true cafile /mosquitto/certs/ca.crt certfile /mosquitto/certs/server.crt keyfile /mosquitto/certs/server.key
[!] NOTE — the mosquitto container runs as a non-root mosquitto user internally, which can't read a 600-permission key file owned by the host user. chmod 644 server.key ca.key before mounting, or the broker fails to start with a cryptic OpenSSL Error: system lib / Permission denied.
Restart everything against the TLS listener and repeat the exact same test — same topic, same-shaped message, only the transport changed:
docker run --rm --network iot-lab-net eclipse-mosquitto \ mosquitto_pub -h mqtt-broker -p 8883 --cafile /certs/ca.crt \ -t High_Tempe -m "This is second alert for High Temperature"
evidence — same grep, port 8883
$ tcpdump -r phase2-tls.pcap -A | grep "High_Tempe\|second alert" (no output — the strings aren't in the capture as plaintext)
Confirming it's actually TLS doing the work, not just a quiet failure — tshark filtered to the TLS records:
$ tshark -r phase2-tls.pcap -Y "tls" -T fields \ -e frame.number -e _ws.col.Protocol -e _ws.col.Info 4 TLSv1.2 Client Hello (SNI=mqtt-broker) 6 TLSv1.3 Server Hello, Change Cipher Spec, Application Data... 8 TLSv1.3 Change Cipher Spec, Application Data 9 TLSv1.3 Application Data 17 TLSv1.2 Application Data
Full handshake (Client Hello → Server Hello → Change Cipher Spec), then every subsequent packet is Application Data — the MQTT PUBLISH is in there, wrapped in TLS 1.3, and opaque to anyone sniffing the wire.

[ mitre att&ck mapping ]
T1040 — Network Sniffing: this is the technique phase 1 leaves wide open — cleartext protocol traffic readable to anyone who can see the segment. TLS on the transport is the direct control against it; it doesn't stop a device from being compromised, but it stops that compromise (or a passive tap elsewhere on the path) from harvesting sensor data, commands, or credentials in transit.
$ lab read LAB-001 --section=takeaways
[ takeaways for network defenders ]
✓Default MQTT (port 1883) is cleartext and typically anonymous — treat any IoT deployment still on it as an active exposure, not a config detail
✓Enabling TLS is a broker-config change plus a cert, not a redesign — the device-side change is just a port number and a CA file
✓Verify the control, don't just enable it — the whole point of this lab is proving the payload is actually unreadable, not trusting that flipping a flag worked
✓Certificate CN matters — self-signed certs still have to match the hostname the client dials, or the handshake fails long before encryption is the issue
