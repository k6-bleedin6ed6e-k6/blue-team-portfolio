[ why I did this ]

I'm working on a cybersecurity degree and I've been doing lab exercises for a while now. Lab traffic is clean by design — the attacker behavior you analyze was constructed to teach you something. That's useful, but it doesn't prepare you for the noise, the weirdness, or the speed of what actually hits an internet-facing server.

I wanted to see it for real. So I stood up an AWS EC2 instance, opened port 22, and waited to see what would happen.

I expected it to take a few days. It didn't.

[ the setup ]

AWS EC2 t3.micro running Ubuntu. Single network interface. Port 22 exposed to the internet with password authentication enabled — just to give attackers something to brute force. Root login was disabled, but I created a standard user account. I wasn't trying to capture credentials; I was trying to capture log data.

/var/log/auth.log was the primary source. I had Splunk Cloud set up to ingest it so I could write SPL queries against what came in.

I did not have fail2ban running initially. That was a mistake I fixed after this hunt. More on that below.
[ what happened ]

Within hours of the instance going up, SSH brute force attempts started showing up in auth.log. On 2026-04-03, I identified a sustained campaign starting around 10:20 UTC: a single IP address attempting authentication at roughly 12 attempts per minute, maintaining 4 parallel connections throughout.

That's not a probe. That's deliberate, sustained brute force. The attacker wasn't checking if SSH was open — they already knew. They were trying to get in.

[ detection — SPL query ]

I wrote this search to surface the pattern from auth.log. The key was finding IPs that exceeded a threshold of attempts per minute — 8 felt like a reasonable floor to separate brute force from background noise.

Splunk SPL — SSH Brute Force Detection
index=linux_syslog sourcetype=syslog
| regex _raw="pam_unix\(sshd:auth\): authentication failure"
| bin _time span=1m
| stats count as attempts by _time, src_ip
| where attempts > 8
| sort -attempts

The results were clear. One IP. Consistent. Sustained. The threshold of 8/min was high enough to filter normal noise but low enough to catch this attacker before they had much time to work.

[ the IP — where it got interesting ]

I ran 20.97.163.85 through AbuseIPDB. 247 reports. I assumed that confirmed it.

Then I looked at the confidence score: 0%.

I didn't understand that at first. How can 247 reports produce zero confidence? I went back and looked at the ASN.

IP20.97.163.85
ASNAS8075 — Microsoft Corporation
HostingMicrosoft Azure
AbuseIPDB Reports247
AbuseIPDB Confidence0% — shared cloud infrastructure

Azure IP ranges are shared. The same IP block can host thousands of different customers — legitimate businesses, researchers, Microsoft's own services, and yes, attackers who spun up an Azure VM to use as a launch point. AbuseIPDB knows this. When it sees high report volume but shared infrastructure, it suppresses the confidence score because blocking that IP could also block legitimate traffic from the same range.

So: 247 people reported it. 0% confidence to act on it as a standalone indicator. That felt contradictory until I understood the reasoning.

Cloud provider IPs — AWS, Azure, GCP — require more context than a report count. The behavior in your own logs is the most reliable signal you have. External reputation is a supporting data point, not the conclusion.
[ what I learned ]
  • IP reputation is not binary. A high report count and a low confidence score can coexist. Context — especially ASN and infrastructure type — changes what a report count means.
  • Behavioral evidence beats point-in-time reputation. 12 attempts per minute sustained over several minutes is attack behavior. I don't need an external database to confirm that. My own logs confirm it.
  • Fail2ban should have been day-one config. I didn't have it running when this attack came in. The attacker got to run 12 attempts per minute for longer than they should have. After this, I installed fail2ban with the sshd jail active. Simple. Should have been first.
  • Real attack data comes in fast. I expected days before anything showed up. It was hours. Any internet-facing SSH port will attract attention almost immediately. That's worth knowing before you build something you think is "just for a minute."
  • SPL thresholds need iteration. My 8-attempts-per-minute threshold worked here, but I set it somewhat arbitrarily. A real detection engineer would tune that threshold against a baseline. I'm working on building that baseline for my honeypot traffic over time.
[ what's next ]

I want to build time-series data from the honeypot — does the same IP come back? Do attacks cluster around certain hours? What's the ratio of credential spraying (many usernames, few passwords) versus pure brute force (one username, many passwords)?

I also want to add geolocation and ASN tagging to the SPL pipeline so I can look at attack distribution over time without having to manually check each IP.

And geoblock. Probably geoblock.

This was five minutes of real log analysis that taught me more about IP reputation and behavioral detection than weeks of lab scenarios. I'd recommend it to anyone learning blue team — stand something up, open a port, and watch what comes in.
